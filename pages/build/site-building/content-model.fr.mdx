# Modélisation de contenu

## Commencez par un MVP

![Produit Minimum Viable](/assets/images/docs/mvp-minimum-viable-product-lean-startup.png)

Drupal est puissant et vous pouvez obtenir rapidement un modèle de contenu complexe et des fonctionnalités en plus de cela.  
Cela peut donc être tentant de livrer rapidement le produit complet. À la place, essayez d'itérer et de suivre les [principes Agile](http://agilemanifesto.org/) (SCRUM, Lean, ...).

## Avoir un bon plan

La création de types de contenu, de vocabulaires et de relations à la volée mènera à une interface utilisateur encombrée et à une structure de données non maintenable.

Voici une méthodologie vraiment simple qui peut s'appliquer à la plupart des projets :

1.  Discutez des règles métier avec le product owner
2.  Discutez avec les personnes qui utiliseront Drupal en tant qu'éditeurs de contenu
3.  Utilisez ce qui vous convient bien pour représenter le modèle de contenu : [carte mentale](https://fr.wikipedia.org/wiki/Carte_heuristique), [diagrammes UML](https://fr.wikipedia.org/wiki/Langage_de_modélisation_unifié) (**cas d'utilisation** et **diagramme de séquence** pour les fonctionnalités et **diagramme de classe** pour les entités sont plus que suffisants, pas besoin d'utiliser plus ici).
4.  Validez votre modèle avec du contenu réel
5.  Itérez, testez et améliorez

## Évitez de définir trop de types de contenu

Soyez honnête avec la réalité des éditeurs de contenu, si l'objectif est de publier quelques nouvelles et emplois par an, ne créez pas deux types de contenu distincts, le type de contenu Article par défaut avec une référence à un vocabulaire Article type avec des termes d'emploi et de nouvelles devrait être suffisant dans la plupart des situations.

## Obtenez un aperçu de la configuration du système de traduction

Il peut être une tâche ardue de revoir le statut de traduction par type de contenu.
Heureusement, un aperçu complet de la configuration des champs par type d'entité est disponible ici : **/admin/config/regional/content-language**

## Traduction de référence d'entité

Les références d'entités sont définies via des champs.  
_Exemple : un type de contenu Article peut faire référence à un terme de taxonomie de Type d'Article. Ou un Article peut faire référence à un type de contenu Organisation._  
  
La plupart du temps, vous n'aurez pas besoin de traduire les références, la seule raison est de définir des valeurs différentes par langue.

## Direction dans les relations d'entités

Prenons un cas d'utilisation courant : vous voulez faire une référence entre deux types de contenu : _Article_ et une _Organisation,_ pour indiquer que l'organisation est l'auteur original de la publication.

En gros, vous avez deux options

1.  Définir un champ Organisation dans le type de contenu Article
2.  Définir un champ de référence d'entité Article dans le type de contenu Organisation

**Première solution**  
Avantages : Rendu et filtrage faciles. Vous aurez probablement moins d'Organisations que d'Articles donc vous pouvez exposer votre champ via ...  
Inconvénients : L'ordre doit être défini par le poids.

**Deuxième solution**  
Avantages : L'ordre peut être effectué par glisser-déposer.  
Inconvénients : Vous aurez probablement plus d'Articles que d'Organisations, donc vous devrez exposer le champ via l'autocomplétion.

## Changer la multiplicité d'un champ

Même si vous avez cet avertissement, vous pouvez toujours changer la multiplicité lorsque des données sont définies pour le champ.

![Changer le nombre de valeurs d'un champ](/assets/images/docs/drupal-change-field-multiplicity.png)

Sachez que **la multiplicité est partagée entre les instances de champ.**  
  
_**Exemple :** vous avez un champ Tags (field_tags) qui est défini sur le type de contenu Article avec une multiplicité (nombre de valeurs) illimitée.  
Vous définissez un nouveau type de contenu Event et partagez ce champ field_tags. Ensuite, vous définissez la multiplicité à 1 valeur. L'effet secondaire est que le champ Tags n'a maintenant qu'une seule valeur possible._

Notez que **les autres paramètres** ne sont pas partagés entre les instances de champ.

_**Exemple** : pour un champ image, vous pouvez définir des répertoires de fichiers différents pour le stockage._
