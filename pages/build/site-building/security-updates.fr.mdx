# Mises à jour de sécurité

L'[équipe de sécurité](https://www.drupal.org/drupal-security-team/general-information) publie des annonces de sécurité _hebdomadaires_, la plupart du temps le jeudi.
N'oubliez pas de le mentionner si vous prévoyez une maintenance.
Ces annonces concernent le noyau Drupal et les projets contribués (modules, thèmes, distributions).

Les annonces de sécurité sont décrites comme suit :

*   Niveau : [Définition des niveaux de risque de sécurité](https://www.drupal.org/drupal-security-team/security-risk-levels-defined)
*   Description

[Page des avis de sécurité](https://www.drupal.org/security)

## Comment être notifié(e) ?

Le module principal _Gestionnaire de mises à jour_ est installé par défaut et envoie des notifications par e-mail + sur votre site lorsque nécessaire, en fonction de votre version de base et des projets installés.

Si vous désactivez ces notifications, assurez-vous de recevoir les mises à jour :

*   Suivez [Drupal Security sur Twitter](https://twitter.com/drupalsecurity)
*   Abonnez-vous à la newsletter de sécurité lors de la création ou de la modification de votre [compte Drupal](https://www.drupal.org/user/register)

## Dois-je vraiment mettre à jour à chaque fois ?

Eh bien, lisez simplement sur [la faille de sécurité Drupageddon](https://www.drupal.org/drupalsa05FAQ), quelques heures après son annonce, les sites qui n'ont pas appliqué le correctif ont été piratés.

Honnêtement, l'injection SQL (le problème de sécurité dans ce cas) est très rare et aussi [un risque de sécurité sur 25](https://www.drupal.org/SA-CORE-2014-005).
Mais il est toujours judicieux de garder votre site à jour, même s'il ne s'agit pas de mises à jour de sécurité. Voici quelques arguments pour vous, votre client ou votre responsable :

*   Un site qui reçoit des mises à jour régulières n'a pas besoin d'une refonte complète, il évolue naturellement et suit les mises à jour des API.
*   Le client ne devrait pas avoir à accepter les mises à jour de sécurité, cela fait également partie d'un processus normal de maintenance logicielle.
*   Cela devrait être inclus par défaut dans le contrat avec votre client.
*   Drupal est livré avec des tests automatisés, donc même s'ils ne remplacent jamais un test manuel, ils réduisent quand même la charge de devoir tester manuellement le site après chaque mise à jour.
*   Pour le code personnalisé, vous devriez écrire des tests unitaires qui accélèrent les tests après la mise à jour.

Dans tous les cas

*   Lorsque les annonces de sécurité sont publiques, examinez le risque de sécurité et lisez attentivement la description pour évaluer si vous devez l'appliquer immédiatement ou si vous pouvez planifier une maintenance planifiée avec votre client.
*   Si vous ne pouvez pas mettre à jour, pour quelque raison que ce soit, appliquez au moins un correctif.
*   Préparez une sauvegarde de vos fichiers, de votre code et de votre base de données (mais cela devrait déjà être le cas avec une solution automatisée).

Certaines [entreprises d'hébergement dédiées à Drupal](/deploy/drupal-hosting-for-evaluation-and-production) fournissent également des services de sécurité supplémentaires, mais la mise à jour de sécurité doit toujours être effectuée.
